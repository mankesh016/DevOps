# Week 30: Observability (Logging, Monitoring, and Alerts)

This section focuses on "Observability," the critical practice of tracking and understanding the internal state of your application in a production environment. We utilize **New Relic** (an industry-standard observability platform) to set up a comprehensive monitoring suite.

---

## 📊 The Three Pillars of Observability

1. **Logging:** The practice of recording discrete events, errors, and informational messages generated by the software.
2. **Monitoring:** The continuous observation of system metrics (CPU, Memory, Network) and Application Performance (Response Times, Throughput).
3. **Alerting:** Automated triggers that page developers (via Slack, Email, or PagerDuty) when systems crash or metrics breach predefined thresholds.

---

## 🛠️ Implementation Highlights

### Infrastructure vs. Application Monitoring

- **Infrastructure Monitoring:** We run a Dockerized New Relic agent directly on our AWS EC2 instance. This provides real-time visibility into the hardware (CPU % usage, Disk I/O, Network Traffic) using NRQL (New Relic Query Language).
- **Application Performance Monitoring (APM):** We instrument our Node.js code using the `newrelic` npm package. By running the app with `node -r newrelic index.js`, New Relic hooks into the V8 engine and Express router to track the exact execution time of individual HTTP endpoints.

### Advanced Logging with Winston

Instead of relying on fragile `console.log` statements, we use **Winston** to handle structured, leveled logging.

- **Log Levels:** Strict categorization (e.g., `error`, `warn`, `info`, `debug`).
- **Transports:** Winston allows logs to be routed simultaneously to multiple destinations. For example, `info` logs might print to the terminal console, while `error` logs are automatically written to a persistent `error.log` file or sent to a remote database.
- _Golden Rule:_ Never store logs purely locally on an ephemeral EC2 machine; if the server crashes, the diagnostic logs crash with it.

## 🧠 Engineering Concept: The Flaw of "Averages"

When measuring application response time, looking at the "Average" (Mean or Median) is a DevOps anti-pattern. A few extremely fast or extremely slow requests can wildly skew the average, hiding terrible user experiences.

**The Solution: Percentiles**
Modern engineering teams optimize for **p95** and **p99** metrics.

- **p95 (95th Percentile):** Indicates that 95% of all requests are faster than this given number. It allows you to ignore the absolute worst 5% of requests (which are often uncontrollable network anomalies) and guarantee a smooth experience for the vast majority.
