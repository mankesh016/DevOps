## Observability: Logging, Monitoring, and Alerts (New Relic)

This class introduces the core concepts of system observability, focusing on setting up logging, monitoring, and alerting using New Relic. _Note: Smaller companies often skip monitoring entirely, but it is crucial for scaling. Next week will cover Prometheus, Grafana, and Kubernetes._

### 1. The Three Pillars of Observability

- **Logging:** Refers to the data points generated by software applications and systems. It is the practice of recording events and messages.
  - _Crucial Rule:_ Never store logs exclusively on the same machine where the application runs; if the machine dies, the logs are gone forever. You should log remotely.
- **Monitoring:** Involves the continuous observation of a system to ensure it is operating correctly and efficiently. Common metrics tracked include CPU usage, Memory usage, Disk I/O space, Network traffic, and Application Performance.
- **Alerts:** Triggers that notify the team when specific conditions are met, such as a system going down, CPU usage spiking above a certain point, or the error count rising rapidly. These are usually tied to On-Call Engineers via platforms like PagerDuty.

#### Popular Tools

- **Paid Services:** AWS CloudWatch, Datadog, New Relic.
- **In-House / Self-Hosted:** Prometheus + Grafana + Loki.

### 2. Infrastructure Monitoring with New Relic

To monitor the underlying EC2 instance (the server itself), you install the New Relic Agent.

1. SSH into your EC2 Linux machine.
2. Install Docker. _(Note: Mac doesn't natively support `network=host` in Docker easily, but Linux does, which the agent uses)._
3. Install the New Relic client via Docker.
4. The default dashboard will instantly start showing: CPU usage (1min, 5min, 10min), Memory usage, Storage, Network traffic, Disk usage, and Load averages.

**NRQL (New Relic Query Language):**

- Similar to SQL (e.g., `SELECT * FROM SystemSample`).
- Features unique keywords like `TIMESERIES AUTO` and `FACET` (used for distinct rows/grouping).

### 3. Application Performance Monitoring (APM)

To monitor the actual Node.js/Express application code, you use APM.

1. Install the package: `npm install newrelic --save`.
2. Add `import "newrelic"` at the very top of your `index.js` file.
3. To start the app, do _not_ run `node index.js`. Instead, pass the configuration via environment variables and require the module:
   ```bash
   NEW_RELIC_APP_NAME="my-app" NEW_RELIC_LICENSE_KEY="123***" node -r newrelic index.js
   ```

````

*Load Testing:* You can test the APM by sending fake traffic using the `loadtest` package.

```bash
npm install -g loadtest
loadtest -c 10 --rps 200 http://localhost:3000/

````

_This will cause a visible spike on your New Relic dashboard._

### 4. Advanced Logging with Winston

Winston is a versatile Node.js logging library that allows you to control exactly where logs are saved (Console, Files, remote databases).

**Log Levels (from most severe to least):** 0. `error`: "Failed to fetch"

1. `warn`: "Deprecated API, use V2"
2. `info`: "Server started on port 3000"
3. `http`
4. `verbose`
5. `debug`: "Cache hit!" (Useful for local development)
6. `silly`

**Winston Setup Example:**

```javascript
import winston from "winston";

const logger = winston.createLogger({
  level: "info",
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json(),
    winston.format.prettyPrint(),
  ),
  transports: [
    new winston.transports.File({ filename: "error.log", level: "error" }),
    new winston.transports.Console(),
  ],
});

logger.error("Failed to fetch");
logger.info("Server running at 3000");
```

### 5. Setting Up Alerts & Workflows

You can configure New Relic to trigger alerts based on specific policies.

- **Examples of Policies:** CPU > 30% for 5 minutes, Memory > 50%, Storage > 70%.
- **Workflows/Channels:** When an alert triggers, it can notify the team via Slack, Microsoft Teams, Email, Webhooks, ServiceNow, PagerDuty, or AWS EventBridge.

### 6. Metrics Theory: Evaluating Response Time

When monitoring Response Time, tracking the "Average", "Mean", or "Median" is often misleading.

**The Solution: Percentiles (p95 and p99)**

- Percentiles measure the worst-case response times while ignoring extreme anomalies.
- **p95:** The response time experienced by the 95th percentile of users. You ignore the absolute worst 5% (anomalies) and optimize for everyone else.
- **p99:** You ignore only the worst 1% of bad traffic/anomalies.
